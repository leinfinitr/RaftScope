# Raft 演示指南

## 项目定位

这个仓库是一个运行在浏览器里的 Raft 可视化演示项目。它不是生产级 Raft 实现，也不是完整的论文复现，而是把选举、心跳、日志复制、故障与恢复等核心过程做成了可观察、可暂停、可回放的交互式动画。

如果你是为了课程展示或论文讲解使用它，可以把它理解成：

- 一个五节点固定集群的 Raft 状态机模拟器。
- 一个能把 `RequestVote`、`AppendEntries`、日志变化和 term 变化直接画出来的教学工具。
- 一个带有两段脚本化场景的课堂演示页。

## 运行方式

### 1. 获取依赖

这个仓库依赖子模块提供前端组件。第一次克隆后先执行：

```bash
git submodule update --init --recursive
```

### 2. 打开页面

任选一种方式：

- 直接在浏览器打开根目录下的 `index.html`
- 或者在仓库根目录启动一个静态服务器，再访问对应地址

例如：

```bash
python -m http.server 8000
```

然后在浏览器里打开 `http://localhost:8000/`。

如果你只是本地演示，直接打开 `index.html` 通常就够用；如果你想避免浏览器对本地文件策略的影响，建议使用静态服务器。

## 仓库结构

下面这些文件是理解项目最关键的部分：

- `index.html`
  页面骨架，包含拓扑图、日志区域、时间轴、速度条、帮助弹窗和两个脚本化演示按钮。
- `raft.js`
  核心 Raft 模拟逻辑，定义服务器状态、选举规则、`RequestVote` / `AppendEntries` 处理、日志复制、提交推进，以及两个内置演示场景。
- `script.js`
  前端交互和渲染逻辑，负责把服务器、消息、日志、时间轴、右键菜单和快捷键连接起来。
- `state.js`
  时间线状态管理，支持回放、拖动、保存检查点和在当前时间点分叉。
- `util.js`
  克隆、比较、计数、圆形布局等通用工具函数。
- `style.css`
  页面样式和 SVG 颜色规则。

下面这些目录主要是前端依赖：

- `bootstrap-3.1.1/`
- `jquery/`
- `bootstrap-slider/`
- `bootstrap-contextmenu/`

## 界面怎么看

页面大致可以分成五个区域：

### 1. 顶部按钮

- `测试复杂场景`
  自动执行一段包含 leader 切换、节点宕机、集群失去多数、恢复后重新追平日志的脚本。
- `提交规则演示`
  自动构造冲突日志和 leader 变更，用来演示未提交日志如何被新 leader 的日志覆盖。

### 2. 日志输出区

页面上方的文本框会输出脚本化场景的步骤说明，适合上课时配合口头讲解。

### 3. 中间的五节点拓扑图

- 五个圆分别代表五个服务器。
- 颜色表示当前 term。
- 红色服务器标签表示当前 leader。
- 圆环上的弧线表示该节点距离下一次 election timeout 还有多久。
- 圆内的小点表示 candidate 当前拿到或正在收到的投票情况。

### 4. 右侧日志区

每一行对应一个服务器当前的日志。

- 方块里的数字是该日志项所属的 term。
- 实线边框表示已提交日志。
- 虚线边框表示未提交日志。
- leader 视图里还会显示每个 follower 的 `matchIndex` 和 `nextIndex`。

### 5. 时间轴和速度条

- 点击时间图标可以暂停或继续。
- 拖动时间轴可以回到过去的状态。
- 速度条可以把模拟放慢，方便逐帧讲解消息流转。

## 这个项目实际演示了哪些 Raft 功能

### 已实现并且可以直接观察的部分

- Follower、Candidate、Leader 三种角色切换。
- 随机选举超时触发的新一轮选举。
- `RequestVote` 请求和应答。
- 基于多数票的 leader 选举。
- leader 周期性发送 `AppendEntries` 作为心跳。
- leader 接收客户端请求后追加日志。
- 日志逐个条目复制到 follower。
- follower 遇到冲突日志时删除冲突后缀并接受 leader 日志。
- leader 根据 `matchIndex` 推进 `commitIndex`。
- 节点故障、恢复、消息丢弃对选举和复制过程的影响。
- 时间线回放与分叉，适合课堂上重复观察某一瞬间。

### 适合课堂解释但需要口头补充的部分

- 日志“新旧程度”会影响投票授权。
- 新 leader 会通过回退 `nextIndex` 找到与 follower 一致的日志前缀。
- 只有当前 term 的日志在多数复制后才会驱动 leader 推进 `commitIndex`。

### 没有实现或做了简化的部分

- 没有 `InstallSnapshot`，也没有日志压缩。
- 没有成员变更。
- 没有真实业务状态机，客户端请求只会追加一个占位值到日志里。
- 集群规模固定为 5 个节点。
- `stop` / `resume` 是简化后的崩溃恢复模型，不等同于完整的持久化存储实现。

## 推荐的演示顺序

如果你需要做一次 5 到 10 分钟的课堂展示，可以按下面的顺序来：

### 第一段：先让大家看懂界面

1. 打开页面后先不要急着点按钮。
2. 说明五个圆代表五个节点，右侧是每个节点的日志。
3. 说明颜色代表 term，红色标签代表 leader。
4. 点击时间按钮暂停，让大家看见消息是如何沿着圆环飞行的。

### 第二段：手动演示一次正常选举和复制

1. 等待系统自然选出一个 leader，或者在某个节点上执行 `time out`。
2. 按 `C` 或对 leader 右键选择 `request`，提交一个客户端请求。
3. 观察 leader 的日志先增长，然后 follower 逐渐复制到相同长度。
4. 暂停后点击某条 `AppendEntries` 消息，展示其中的 `prevIndex`、`prevTerm`、`entries`、`commitIndex`。

### 第三段：演示故障和恢复

1. 对 leader 使用 `network failure` 或 `stop`。
2. 观察新的选举开始，其他节点争夺 leader。
3. 恢复节点后，观察它如何重新回到 follower 并追上日志。

### 第四段：用快捷键调整超时

- 按 `T` 可以把超时拉开，降低 split vote 的概率。
- 按 `A` 可以把超时对齐，更容易出现竞争选举。

这一段适合讲“为什么 Raft 依赖随机超时来避免频繁平票”。

### 第五段：播放内置脚本化场景

先跑 `测试复杂场景`，再跑 `提交规则演示`。前者适合讲集群故障、恢复和重新达成一致，后者适合讲冲突日志覆盖。

## 手动操作说明

### 点击服务器

点击任意服务器会弹出详情框，可以看到：

- `state`
- `currentTerm`
- `votedFor`
- `commitIndex`
- `electionAlarm`
- 面向每个 peer 的 `nextIndex`
- 面向每个 peer 的 `matchIndex`
- 是否已经收到该 peer 的投票

这个弹窗很适合在讲到 leader 日志复制时解释 `nextIndex` 和 `matchIndex` 的作用。

### 右键服务器

右键服务器会出现操作菜单：

- `network failure`
  把节点切到 `stopped`，模拟网络故障。
- `network recovery`
  从故障中恢复，重新作为 follower 参与选举。
- `stop`
  模拟停机。实现里会把该节点日志截断到 `commitIndex`。
- `resume`
  从 `stop` 状态恢复。
- `time out`
  让该节点立刻触发一次选举。
- `request`
  如果该节点当前是 leader，则向它提交一个客户端请求。

### 点击消息

点击消息可以查看详情：

- 对 `RequestVote`，可以看到 `lastLogIndex`、`lastLogTerm` 或 `granted`
- 对 `AppendEntries`，可以看到 `prevIndex`、`prevTerm`、`entries`、`commitIndex` 或 `success`、`matchIndex`

### 右键消息

右键消息后可以选择 `drop`，手动丢弃这条尚未送达的消息。这是构造网络异常最直接的方法。

### 常用快捷键

- `Space` 或 `.`
  暂停 / 继续。
- `?`
  打开帮助。
- `C`
  向当前最高 term 的 leader 提交请求。
- `R`
  重启当前最高 term 的 leader。
- `T`
  拉开超时，减少 split vote。
- `A`
  对齐超时，增加 split vote 的机会。
- `B`
  恢复全部服务器。
- `F`
  以当前时间点为新分支，丢弃未来时间线。

## 两个内置脚本化场景怎么讲

### 场景一：`测试复杂场景`

这个按钮会自动执行一组多阶段操作，适合说明“在连续故障和恢复之后，系统如何重新形成多数并追平日志”。

大致过程是：

1. 让 Server 1 成为 leader。
2. 向 Server 1 连续提交两个请求，并确保它们复制到多数节点。
3. 让 Server 1 宕机，促使 Server 2 成为新的 leader。
4. 继续制造节点故障，让集群暂时失去多数。
5. 恢复一部分节点，让 Server 5 成为新的 leader。
6. 向 Server 5 提交新请求。
7. 恢复之前故障的节点，观察它们重新追上当前 leader 的日志。

演示时建议重点强调三件事：

- leader 宕机后会重新选举。
- 没有多数时，新的日志无法稳定提交。
- 节点恢复后会由当前 leader 把日志重新补齐。

注意：

- 这个脚本内部一共有 12 个阶段，但日志文本里最后两个阶段都写成了“步骤11”，这是现有脚本里的编号小瑕疵。

### 场景二：`提交规则演示`

这个按钮会构造一个更适合讲“冲突日志覆盖”的场景。

它做的事情可以概括为：

1. 先让 Server 1 成为 leader 并提交一条日志。
2. 再制造一次“只有部分节点收到新日志”的情况，形成不一致的未提交后缀。
3. 让 leader 反复切换，分别在不同节点上追加新日志。
4. 最后让新的 leader 把自己的日志复制给所有节点。

这个场景最适合讲下面这条 Raft 规则：

- 未提交的日志并不保证最终保留。
- 当新的 leader 拥有不同的日志前缀时，follower 会删除冲突后缀并接受 leader 的日志。

如果你想把这段讲得更清楚，可以在场景运行过程中暂停，然后重点观察：

- `AppendEntries` 的 `prevIndex` / `prevTerm`
- follower 在冲突发生后日志长度的变化
- leader 视图中的 `nextIndex` 和 `matchIndex`

## 已知限制与注意事项

### 1. 这是教学演示，不是完整工程实现

它非常适合展示 Raft 的核心流程，但不适合作为生产系统的实现参考。

### 2. `L` 快捷键对应的预设场景不要使用

帮助弹窗里写了 `L` 可以进入一个日志复制预设场景，但 `raft.setupLogReplicationScenario()` 内部调用了未定义的 `raft.restart`。也就是说，这条帮助文案和实际代码不一致，文档中不把它当作可用功能。

### 3. `stop` 和 `network failure` 不完全一样

- `network failure` 更像是让节点失联。
- `stop` 会额外把日志截断到 `commitIndex`，因此更接近“简化后的崩溃恢复”。

演示时最好说明这是项目作者为了教学效果做的简化。

### 4. 没有快照和成员变更

如果你的课堂内容要覆盖快照安装、日志压缩或联合共识，这个仓库本身并没有把这些内容做出来，需要额外口头补充。

## 一个简短的课堂演示脚本

如果你只想照着讲，可以直接用下面这段：

1. 这是一个五节点的 Raft 可视化页面，左边是节点和消息，右边是每个节点的日志。
2. 我们先看 leader election，红色标签代表 leader，颜色代表当前 term。
3. 现在给 leader 提交一个请求，可以看到 leader 先追加日志，再通过 `AppendEntries` 复制给 follower。
4. 当多数节点都复制完成后，这条日志就会被提交。
5. 接下来我让 leader 故障，大家可以看到新的选举开始，系统会选出新的 leader。
6. 最后我运行内置的规则演示，观察未提交冲突日志如何被新 leader 的日志覆盖。
